# 易宿酒店预定平台项目报告

## 1. 项目介绍
易宿酒店预定平台是一个集成了PC端管理后台与移动端（M端）用户预订功能的综合性酒店服务系统。项目采用前后端分离架构，旨在为用户提供便捷的酒店搜索、预订体验，同时为商家和管理员提供高效的酒店管理、审核与经营分析工具。

项目主要分为两个核心端：
- **PC端**：主要面向商家（Merchant）和管理员（Admin）。商家可进行酒店创建、管理、订单查看及通过AI助手获取经营指导；管理员负责酒店入驻审核及平台监管。
- **M端**：面向终端消费者。提供酒店推荐、搜索、详情查看、下单支付、订单管理及AI客服助手等功能。

## 4. 个人报告1 - 组长

### 4.1 技术解析

#### 4.1.1 总体架构
项目采用前后端分离架构：
- **PC 端前端**：基于 **React** 框架，使用 Tailwind CSS 进行样式开发。
- **M 端前端**：基于 **Taro** 跨端框架（React 语法），支持编译为微信小程序及 H5。
- **后端**：基于 **Node.js + Express** 框架，数据库采用 **PostgreSQL**，ORM 框架使用 **Sequelize**。

#### 4.1.2 PC 端技术详解
##### 后端技术架构
- **分层架构**：严格遵循 `controllers`（控制器）、`services`（业务逻辑）、`models`（数据模型）、`routes`（路由）、`middlewares`（中间件）的分层设计，确保代码高内聚低耦合。
- **数据库设计与初始化**：基于 PostgreSQL + Sequelize 完成数据库建模与初始化，围绕“用户-酒店-房型-订单”主链路进行扩展，共设计了 26 张核心业务表（包含若干多对多中间表），并通过外键与索引保证数据一致性与查询性能。
  - **用户域**：`users` 作为统一账号表（通过 `role` 区分商户/管理员/移动端用户），`user_profiles` 用于扩展资料（昵称、性别、生日、头像等），`verification_codes` 存储短信验证码并支持过期/已使用状态控制。
  - **酒店域**：`hotels` 存储酒店基础信息与状态流转（草稿/待审核/通过/驳回等），`hotel_policies` 维护酒店政策（取消、支付、儿童、宠物），`audit_logs` 记录管理员审核动作，`hotel_history` 用于关键字段变更留痕与版本追踪。
  - **房型与定价域**：`room_types` 维护房型信息，`room_prices` 以“房型 + 日期”建模实现价格日历，`room_policies` 与 `room_tags` 用于承载政策与标签等扩展配置。
  - **交易与互动域**：`bookings` 记录订单主信息并关联用户/酒店/房型，`hotel_reviews` 记录评价（支持评分、文字与图片），`favorites` 维护收藏关系，`messages` 承载站内通知消息。
  - **字典与多对多关系**：`facilities`、`services` 作为酒店/房型配置字典表，通过 `hotel_facilities`、`hotel_services`、`room_facilities`、`room_services` 四张中间表实现灵活的多对多配置。
  - **结构化字段策略**：对地址位置 JSON 字段保存结构化数据，减少表拆分成本并便于前端直接渲染。
  - **初始化脚本**：提供数据库重置脚本，通过 `sequelize.drop()` + `sequelize.sync()` 一键完成表结构重建，便于本地联调与测试数据初始化。
- **接口设计与模块划分**：在 `routes/pc` 下按业务模块拆分接口（`auth` / `hotel` / `admin` / `user` / `chat`），配合中间件完成参数校验与权限控制，确保接口职责清晰、便于迭代维护。
  - **认证模块（auth）**：提供账号校验、验证码发送、注册、登录、忘记密码、重置密码等接口，统一通过校验中间件拦截非法参数。
  - **酒店模块（hotel）**：提供酒店创建、编辑、列表、详情、审核状态查询、删除等接口，均要求携带 Token，防止越权操作。
  - **管理员模块（admin）**：提供酒店审核列表、酒店详情、批量审核等接口，除 Token 校验外，额外通过 `requireAdmin` 实现角色鉴权。
  - **用户模块（user）**：提供个人信息查询/更新与消息列表拉取等接口，保证用户侧数据的读写安全。
- **Utils 工具库封装**：
  - **验证码生成**：封装随机验证码生成函数。
  - **阿里云短信服务**：集成 `@alicloud/dypnsapi`，封装短信发送函数，用于注册、找回密码等场景。
  - **认证函数**：封装手机号校验、角色用户名生成等逻辑。
  - **酒店查询**：封装复杂的酒店筛选与查询构造函数。
- **环境配置 (.env)**：统一管理敏感配置，包括 `PORT`、`DATABASE_URL`、阿里云 `accessKeyId/Secret`、`JWT_SECRET`、智谱 `AI_API_KEY` 等。
- **安全认证与鉴权 (JWT)**：使用 `bcryptjs` 对密码进行加密存储；使用 `jsonwebtoken` 生成 Token（携带 `userId/phone/role` 等信息与过期时间），并通过自定义 `authenticateToken` 中间件在后续接口中统一校验 `Authorization: Bearer <token>`，实现会话过期与角色越权拦截。
- **图片上传与静态资源策略**：针对酒店主图、房型图片等上传场景，后端实现“Base64 转文件并落盘”的处理逻辑，将图片写入前端 `public` 目录（如 `main_image`、`room_image`），接口仅返回可访问的 URL 列表，避免将大体积 Base64 数据写入数据库或在接口中频繁传输。
- **智能 Agent 助手（商户经营/平台操作）**：基于智谱 GLM 的对话能力 + Function Calling 机制实现“可检索、可调用工具”的智能问答。
  - **实现逻辑**：用户提问 -> LLM 决策是否调用工具 `knowledge_base_search` -> 后端检索本地知识库并返回上下文 -> LLM 结合工具结果生成更专业的最终回答。
  - **目的**：为商户提供平台操作指引与经营建议，降低学习成本并提升使用效率。
- **消息通知系统**：设计了消息表与轮询机制，当酒店审核状态变更（通过/驳回）时，实时写入消息表，商户端可即时获取通知反馈。
- **自动化测试**：`tests` 目录下包含针对各个核心接口的测试脚本，确保接口交付质量。

##### 前端技术架构
- **模块化设计**：主要划分为 `auth`（认证）、`merchant`（商户）、`admin`（管理员）三大功能板块。
- **API 统一封装**：在 `utils` 目录下封装 `fetch` 请求，自动在 Header 中注入 `Authorization: Bearer <token>`，并统一处理 401、403 等错误状态。
- **地图服务集成**：在“创建酒店”页面，引入 `amap-jsapi-loader` 加载高德地图 JSAPI。
  - **功能**：允许商户在地图上搜索地址或手动拖拽标记点，自动获取经纬度与格式化地址，解决手动输入地址不准确的问题。
- **静态资源管理**：后端接收图片 Base64 后，将其写入前端 `public` 目录并生成访问 URL，既减轻了数据库存储压力，又保证了图片的快速加载。

- **Bug 修复与重构**：
  - **推荐系统修复**：解决了首页搜索栏下方推荐酒店不显示或数据错误的 Bug。
  - **列表页重构**：优化了酒店列表的渲染性能与筛选逻辑，提升滚动流畅度。
  - **详情页重构**：重新设计了酒店详情页的信息结构，使其更符合移动端阅读习惯。
  - **订单流程优化**：修复了订单确认页和详情页的数据加载延迟与状态同步问题，统一了页面 UI 风格，提升用户体验。

### 4.2 主要问题与解决方案

1.  **图片传输与存储效率问题**
    - **问题**：早期方案尝试在接口中直接传递并在数据库存储 Base64 图片数据，实际开发中容易出现内存占用飙升（Node.js Buffer/字符串开销大）、数据库体积快速膨胀、接口响应变慢等问题，尤其在酒店多图、房型多图场景下更明显。
    - **解决**：将图片上传策略调整为“Base64 转文件落盘 + 返回 URL”：
      - 后端将 Base64 解析为二进制并写入静态资源目录（前端 `public` 下的图片目录），生成唯一文件名。
      - 数据库存储与接口返回仅保留图片 URL（支持单图/多图）。
      - 前端展示时直接使用 URL 加载图片，显著降低接口负载与数据库存储压力，并提升页面加载速度。

2.  **酒店地址定位不准**
    - **问题**：仅靠用户输入文字地址，难以在地图上精确展示酒店位置。
    - **解决**：引入高德地图 API，提供“地图选点”功能，强制获取精准经纬度，确保 M 端导航功能的可用性。

3.  **商家操作指引缺失**
    - **问题**：新入驻商家不熟悉后台操作流程。
    - **解决**：开发智能 Agent 助手，结合本地知识库与 LLM，提供 24 小时智能问答与操作引导。

4.  **审核状态通知滞后**
    - **问题**：商家提交审核后无法及时获知结果。
    - **解决**：构建站内消息系统，审核动作触发时自动插入消息记录，商户端通过接口定期拉取或刷新获取最新状态。

  ### 5.2 主要问题与解决方案
1.   **定位不准确问题** ：
     - **问题**：前端定位不准确，总是定位北京。
     - **解决方案**：1. 排查前端定位SDK配置，移除默认经纬度（北京坐标）的硬编码，改为优先调用设备原生定位API。
2. 增加定位失败降级策略：若原生定位失败，自动切换为IP定位获取城市，同时提示用户手动选择位置。

2.   **登录过期不提示问题 ** ：
     - **问题**：登录状态过期仍显示登录。
     - **解决方案**：1. 前端在请求拦截器中统一监听401状态码，触发Token刷新流程；若刷新失败，清除本地存储与全局状态，并弹窗提示“Token已过期，请重新登录”。
2. 后端在Token中嵌入过期时间，前端在页面初始化时主动校验Token有效期，若已过期则直接跳转登录页。

3.   **首页广告不跳转问题 ** ：
     - **问题**：点击banner广告不跳转到详情页。
     - **解决方案** ：
1. 排查广告数据结构，确保后端返回的广告列表中包含合法的跳转类型（如酒店ID、外部链接）与目标ID。
